1. Dissertation Research Framing
1.1 The Gap in EvoAgentX
EvoAgentX provides a five-layer architecture (basic components, agent, workflow, evolving, evaluation) for building and evolving multi-agent workflows. Its workflow layer supports two graph types: WorkFlowGraph (explicit task graphs with custom nodes, edges, conditional branches) and SequentialWorkFlowGraph (auto-inferred linear pipelines). Both treat agents as peers in a flat or sequential topology.
What EvoAgentX does not provide is any concept of organizational hierarchy: supervisor agents that oversee teams, delegation of authority with scope constraints, escalation pathways when a worker agent exceeds its competence, or differentiated roles within a team (manager vs. specialist vs. reviewer). Every agent in vanilla EvoAgentX operates at the same authority level.

The Research Gap
Real-world organizations solve complex problems through hierarchical team structures: managers decompose goals, delegate subtasks to specialists, review outputs, and escalate exceptions. Current multi-agent frameworks, including EvoAgentX, lack this organizational structure. This limits performance on complex tasks that require task decomposition, quality oversight, and coordinated specialization.

1.2 Research Question

Research Question
Does adding hierarchical team-based orchestration (supervisor agents, role differentiation, delegation scoping, and escalation pathways) to a self-evolving multi-agent framework improve task performance on complex, multi-step problems compared to flat workflow orchestration?

1.3 Hypothesis
H1: Hierarchical EvoAgentX (Evolvian) will outperform vanilla EvoAgentX on complex tasks requiring multi-step reasoning and cross-domain coordination (GAIA, HotPotQA), as measured by accuracy/F1 score.
H2: Hierarchical EvoAgentX will show comparable or better performance on structured single-domain tasks (MBPP, MATH), demonstrating that the hierarchy does not add harmful overhead to simpler tasks.
H3: The self-evolution algorithms (TextGrad, AFlow) will produce greater performance gains when applied to hierarchical workflows than flat workflows, because the hierarchy provides more structured optimization targets.

1.4 Contribution Statement
This dissertation makes three contributions:
	•	Architectural contribution: A hierarchical team orchestration layer added to the EvoAgentX framework, introducing supervisor agents, role-based teams, delegation scoping, and escalation pathways.
	•	Empirical contribution: A comparative evaluation on four benchmarks (HotPotQA, MBPP, MATH, GAIA) showing performance differences between flat and hierarchical multi-agent orchestration.
	•	Design contribution: A reusable architecture pattern for hierarchical self-evolving agent teams that can serve as the foundation for enterprise agent management systems.

2.2.4 New Class: HierarchicalWorkFlowGraph
This is the core new graph type. It extends WorkFlowGraph to support teams as first-class nodes, not just individual agents.

Execution flow in a HierarchicalWorkFlowGraph:
	•	Task arrives at the root supervisor (or first team’s supervisor if no root).
	•	Supervisor decomposes the task into subtasks using its LLM. Each subtask is tagged with the skills/scope it requires.
	•	Delegation engine matches subtasks to workers based on the DelegationPolicy (capability matching, load balancing, etc.).
	•	Workers execute their subtasks. If a worker hits an escalation condition, it routes back to the supervisor.
	•	Supervisor reviews outputs (based on review_mode). If quality is insufficient, the supervisor can reassign, provide feedback, or revise the subtask prompt.
	•	Cross-team handoff: When the current team’s work is complete, results pass to the next team in the team_graph via the inter_team_protocol.
	•	Final aggregation: The root supervisor (or last team’s supervisor) assembles the final output.

3. Evaluation Plan (Proving Hierarchy Beats Flat)
3.1 Experimental Design
You will run the same benchmarks that the EvoAgentX paper used, so your results are directly comparable to their published numbers. This is critical for the dissertation: you are extending their work and showing improvement.


Experimental Setup
For each benchmark, run three configurations: (A) Vanilla EvoAgentX (flat workflow, their published settings), (B) Hierarchical EvoAgentX WITHOUT evolution (your team structure, but no optimization — isolates the effect of hierarchy alone), (C) Hierarchical EvoAgentX WITH evolution (your team structure + TextGrad/AFlow optimization — shows the combined effect). Compare A vs B to measure the value of hierarchy. Compare B vs C to measure how evolution interacts with hierarchy. Compare A vs C for the full combined improvement.

3.3 Specific Team Structures Per Benchmark
You need to design the hierarchical team structure for each benchmark. These should be motivated by how humans would organize to solve these tasks.

HotPotQA Team Structure:
	•	Supervisor: Research Coordinator — receives the multi-hop question, decomposes it into sub-questions
	•	Worker 1: Retriever Agent — searches for evidence relevant to each sub-question
	•	Worker 2: Reasoning Agent — synthesizes evidence into an answer to each sub-question
	•	Supervisor reviews the sub-answers, identifies gaps, may re-delegate, then produces final answer

GAIA Team Structure:
	•	Root Supervisor: Task Planner — analyzes the complex task, creates a multi-step plan
	•	Team 1 (Research): Web Navigator + Data Extractor — supervised by Research Lead
	•	Team 2 (Analysis): Data Analyst + File Processor — supervised by Analysis Lead
	•	Root Supervisor aggregates outputs from both teams, handles cross-team dependencies

MATH Team Structure:
	•	Supervisor: Math Strategist — reads the problem, decides solution approach, delegates steps
	•	Worker 1: Solver Agent — executes the mathematical steps
	•	Worker 2: Verifier Agent — checks each step for errors (acts as reviewer)
	•	Supervisor makes final decision based on solver output + verifier feedback

MBPP Team Structure:
	•	Supervisor: Architect Agent — reads the specification, designs the approach, delegates
	•	Worker 1: Coder Agent — writes the implementation
	•	Worker 2: Tester Agent — writes tests and validates the code
	•	Supervisor reviews test results, may request revision from Coder if tests fail

3.4 Evaluation Protocol
Follow EvoAgentX’s published protocol exactly for comparability:
	•	Sample sizes: 50 examples for validation, 100 for testing (same as EvoAgentX paper)
	•	Run each configuration 3 times and report mean + standard deviation (for statistical significance)
	•	Use the same LLM (GPT-4o-mini or whichever model your budget allows) across all configurations
	•	For evolution runs: use the same number of optimization iterations as the EvoAgentX paper
	•	Track and report: total LLM API calls (to show overhead cost of hierarchy), total tokens consumed, and wall-clock time


Expected Results Pattern
You should expect: GAIA shows the largest improvement from hierarchy (most complex, multi-domain tasks). HotPotQA shows moderate improvement (multi-hop reasoning benefits from decomposition). MATH and MBPP may show smaller improvement (more structured tasks where hierarchy adds review loops but the core reasoning is still single-agent). If hierarchy hurts on simpler tasks, that is also a valid finding — you can discuss the overhead-vs-benefit tradeoff in your dissertation.
