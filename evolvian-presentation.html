The Base Question
What EvoAgentX doesn't have
EvoAgentX — Flat Topology
All agents operate at the same authority level. No supervision, no escalation, no delegation.
Evolvian — Hierarchical Tree
Supervisors decompose, delegate, review. Workers escalate when stuck. Organizational hierarchy for agents.
"Does adding hierarchical team-based orchestration improve task performance on complex, multi-step problems compared to flat workflow orchestration?"
Research Foundation
"Does adding hierarchical team-based orchestration — supervisor agents, role differentiation, delegation scoping, and escalation pathways — to a self-evolving multi-agent framework improve task performance on complex, multi-step problems?"
H1 — Supported ✓
On HotPotQA, the Warden team achieved acc = 0.65 vs flat's 0.35 — nearly double. Supervisor-directed decomposition and synthesis genuinely improves multi-hop answer discovery.
H2 — Negative Finding
On MATH Level 2–3, all multi-agent configs scored 0.05 vs the single agent's 0.25. Decomposition breaks tight mathematical reasoning chains — a clear boundary condition.
H3 — Inconclusive
Evolution (Config D) worked mechanically — MATH hit 0.20 at step 2 (4×), then overfit and regressed. train_k = 5 is too small for reliable gradient estimates. Needs more data.

Key Finding
F1 is misleading here. Config A's terse guesses score high on token overlap — but it gets 65% of answers wrong. The Warden finds the correct answer 65% of the time, but its explanatory outputs are penalised by F1.
Hierarchy vs Specialisation
Config B (specialisation only) reaches 0.45 acc. Config C adds the Warden and jumps to 0.65 — isolating that coordination and review drive the gain, not just having specialist roles.
Escalation dropped from 60% (n=5) to ~30% after Verifier calibration. When triggered, the Warden's direct access to full context produced better fallback answers.

What changes between generations
Tree topology is fixed. Same Warden + 3 workers, same roles. Only agent descriptions are mutated — the Optimiser LLM rewrites them to be more precise. Names and hierarchy are never changed.
Why it failed here
train_k=5 is too small. The optimiser found signal in 3/3 MATH failures and updated descriptions, but the updates overfit to those 5 examples. Needs train_k ≥ 20 and max_steps ≥ 10 for stable improvement.

→
Operations are team-scoped. You assign agents, the system auto-builds a hierarchy: root supervisor → sub-supervisors → leaf workers. Tree depth scales with team size.
→
Delegation is capability-matched. Sub-tasks are scored against each agent's name, role, specialty, and description. The best-scoring agent wins the assignment.
→
Escalation is rule-based. Workers that hit a condition (confidence low, error count exceeded) escalate back to their supervisor: retry, reassign, or request human input.
→
Every execution is a data point. WorkflowExecution records cost, latency, token usage, quality score — feeding the evolutionary flywheel.
Human → Agent
How knowledge transfers
01
System Prompt
Core instructions that shape the agent's behaviour — injected into every LLM call at runtime. Defines personality, expertise, and constraints. Editable per-agent.
Agent Intelligence
02
Knowledge Base (RAG)
Each agent has a personal knowledge base — chunks of text, URLs, or documents. These are embedded and injected into the prompt context at execution time. No hallucination needed.
Per-Agent RAG
03
Assumptions Inbox
Agents ask clarifying questions instead of hallucinating. Questions surface with priority (low/normal/high/critical), suggested options, and context. Human answers; agent proceeds with certainty.
Grounded Output
Human knowledge flows in · Agent certainty flows out

Research · Near Term
Agent-to-Agent Assumption Loop
Each agent asks clarifying questions not just to humans but to peer agents in the same team — back-and-forth dialogue before acting. Reduces errors, surfaces hidden assumptions early.
