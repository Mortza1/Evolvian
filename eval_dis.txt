\section{Implementation and Contributions} \label{cha:imple}

This section details the implementation of Evolvian. The system realises the
four-module architecture from Section~\ref{sec:method_two}, the Warden agent, Worker agents,
Memory Layer and Evolution engine, as a production integrated system built on
top of EvoAgentx. The implementation is organised into five packages:
(i) a hierarchy module implementing the Warden and governance graph,
(ii) benchmark-specific team configurations for HotPotQA and MATH,
(iii) an evaluation harness supporting four ablation configurations,
(iv) a statistical comparison framework, and
(v) a comprehensive test suite of 99 unit and integration tests.

The system is integrated into the Evolvian platform (a FastAPI backend with a
React/TypeScript frontend) where it powers real-time hierarchical task
execution with Server-Sent Events (SSE) streaming and human-in-the-loop
quality feedback. This chapter describes each component in detail, maps the
implementation to the formal architecture and research questions from
Section~3, and presents the ablation study design used to isolate the
contributions of hierarchy (RQ1), persistence (RQ2), evolution (RQ3), and
human oversight (RQ4).

\subsection{Warden Agent — Hierarchical Governance Implementation} \label{imp:one}

The Warden Agent, described formally in section~\ref{sec:method_two} as the top-level controller governing the hierarchical governance graph G = (V, E, H), is implemented across three modules:

\subsubsection{Data Model Layer (\texttt{hierarchy/team.py})}
\leavevmode\par

This module defines the organisational primitives that encode the governance
graph $G = (V, E, H)$, where $V$ represents worker agents, $E$ their
communication links, and $H$ the supervisory hierarchy. All classes use
Pydantic-based data models extending EvoAgentX's \texttt{BaseModule},
ensuring serialisability and validation.

Table~\ref{tab:data-model} summarises the five core abstractions; all use Pydantic-based models for serialisability and evolutionary mutation.

\centering\small
\begin{tabular}{lp{4.6cm}p{4.6cm}}
\toprule
\textbf{Class} & \textbf{Role in $G=(V,E,H)$} & \textbf{Key fields / strategies} \\
\midrule
\texttt{Team} & One level of $H$; enforces structural invariants & Warden, workers, DelegationPolicy, EscalationRules, ReviewMode \\
\texttt{HierarchicalAgent} & Node $v\in V$; extends EvoAgentX \texttt{Agent} & role (supervisor/worker/reviewer), team\_id, authority\_scope \\

\texttt{EscalationRule} & Condition-action pair in $H$ & NL condition, action (escalate/retry/HITL/fail), max\_retries \\
\texttt{HierarchicalAgentManager} & Index over $G$ & $O(1)$ team/Warden/worker lookup; scope-based discovery \\

\subsubsection{Warden Intelligence Layer (\texttt{hierarchy/supervisor.py})}
\leavevmode\par

The three cognitive components implement graceful degradation throughout. \textbf{\texttt{SupervisorDecomposer}} calls the Warden's LLM to produce a JSON array of subtasks with \texttt{required\_skills}; on any parsing failure it falls back to a single pass-through subtask. \textbf{\texttt{DelegationEngine}} scores workers via capability-match (+2 exact, +1 partial overlap), maintaining assignment counts for load-balancing, and falls back to round-robin when skills are unspecified. \textbf{\texttt{SupervisorReviewer}} calls the Warden's LLM to produce a \texttt{ReviewDecision} with \texttt{approved}, \texttt{revisions\_needed}, and per-worker \texttt{feedback}; when \texttt{ReviewMode.NONE} is set, it aggregates outputs directly. Every LLM failure has a deterministic fallback, ensuring hierarchical execution always produces an output.

\subsubsection{Governance Graph Construction (\texttt{hierarchy/hierarchical\_graph.py})}
\leavevmode\par

This module extends EvoAgentX's \texttt{WorkFlowGraph} to make teams
first-class graph nodes, directly implementing the governance graph
$G = (V, E, H)$ from Equation~1.

The \texttt{HierarchicalWorkFlowGraph} stores team identity inside
existing \texttt{WorkFlowNode} objects by writing a tag string
(\texttt{\_hierarchical\_team\_id:<team\_id>}) into the node's
\texttt{reason} field. This tag-based polymorphism was chosen deliberately
to avoid modifying EvoAgentX's schema, maintaining full upstream
compatibility. At runtime, \texttt{is\_team\_node()} and
\texttt{get\_team\_for\_node()} read the tag to distinguish hierarchical
team nodes from regular flat nodes, enabling polymorphic dispatch in the
execution engine.

The \texttt{from\_teams()} factory method constructs a complete graph from
a list of \texttt{Team} objects: it creates one \texttt{WorkFlowNode} per
team, generates sequential edges $E$ by default (or accepts custom
dependency specifications for DAG topologies), and populates each node's
\texttt{agents} field with the full agent roster for compatibility with
EvoAgentX's scheduling algorithms.

A \texttt{validate\_hierarchy()} method performs structural validation: it
verifies that every team has a Warden, checks for cycles in the team
dependency graph using depth-first search (ensuring $H$ is a valid
hierarchy), and ensures worker names are globally unique across all teams.


\subsection{Execution Engine — The Warden's Runtime} \label{imp:two}

The execution engine (\texttt{hierarchy/execution.py}) is the
largest and most complex component. It overrides EvoAgentX's
\texttt{WorkFlow} to execute team nodes via the full Warden loop,
implementing the lifecycle management described in
Section~\ref{sec:method_two}: ``governing agent lifecycles (spawning,
promotion, and retirement) while dynamically decomposing tasks.''

The \texttt{HierarchicalWorkFlow} class overrides \texttt{async\_execute()} and \texttt{execute\_task()}; team nodes are dispatched to \texttt{\_execute\_team\_node()}, which runs the six-phase Warden loop formalised in Algorithm~\ref{alg:warden-loop} and illustrated in Figure~\ref{fig:warden-loop}.

% Requires: \usepackage[ruled,vlined]{algorithm2e}
\begin{algorithm}[h!]
\small
\caption{\textsc{WardenExecutionLoop}$(T,\,\mathcal{T})$}
\label{alg:warden-loop}
\DontPrintSemicolon
\KwIn{Task $T$;\; team $\mathcal{T}=\{\text{Warden},w_1,\ldots,w_k\}$}
\KwOut{Final synthesised answer}
$\mathbf{S} \leftarrow \text{Warden.Decompose}(T)$\tcp*{JSON subtasks $\{s_i\}$ with required\_skills}
$\text{asgn} \leftarrow \text{DelegationEngine}(\mathbf{S},\mathcal{T})$\tcp*{capability-match scoring}
\ForEach{$s_i \in \mathbf{S}$}{
  $r_i \leftarrow w_{\text{asgn}[i]}.\text{Execute}(s_i)$\;
  \If{\upshape EscalationCheck$(r_i)$\tcp*[f]{empty / conf.$<$30 + error-cite / LLM}}{
    $r_i \leftarrow \text{Warden.DirectSolve}(s_i)$\;
  }
}
$\mathrm{rev} \leftarrow 0$\;
\Repeat{\upshape approved \textbf{or} $\mathrm{rev} \geq 2$}{
  $(\text{approved},\,\text{fb},\,\text{retry}) \leftarrow \text{Warden.Review}(\{r_i\})$\;
  \ForEach{$s_i \in \text{retry}$}{$r_i \leftarrow w_{\text{asgn}[i]}.\text{Execute}(s_i,\,\text{fb}[i])$\;}
  $\mathrm{rev} \mathrel{+}= 1$\;
}
\Return{\upshape Warden.Synthesise$(\{r_i\})$}\;
\end{algorithm}

\paragraph{\textbf{Execution Trace and Persistence.}}
The engine includes an \texttt{ExecutionTrace} system that records every
event (decompose, delegate, execute, escalate, review, revise, handoff)
with timestamps and metadata. These traces implement a component of the
Memory Layer described in Section~\ref{sec:method_two}: new reasoning
traces $\Delta R_t$ are recorded per execution cycle and persisted
alongside benchmark results. The trace data enables post-hoc analysis
of coordination dynamics (supporting qualitative behavioural analysis
from Section~\ref{sec:method_five}) and feeds into the evolutionary feedback loop
(Equation~4).

\paragraph{\textbf{Cross-team handoff.}}
Cross-team handoff is handled automatically through EvoAgentX's
\texttt{Environment} abstraction: upstream team outputs are stored as
\texttt{Message} objects in the shared environment, and downstream
teams read them by parameter name. This means multi-team pipelines,
where multiple Wardens coordinate, work without explicit data routing
code.

\paragraph{\textbf{SSE integration.}}
SSE integration enables real-time frontend visualisation: the execution
engine accepts an optional \texttt{sse\_emit} callback invoked for every
hierarchy event. The Evolvian backend bridges the async execution
engine with a synchronous SSE generator using a background thread with
its own event loop, polling a shared queue every 200\,ms to yield
buffered events to the client. This enables the human operator to
observe and intervene in the Warden's decisions in real time (RQ4).


\subsection{Auto-Build — Dynamic Team Design} \label{imp:three}

The auto-build layer enables fully automatic team construction from a
plain-text task description. A single LLM call receives a structured
prompt requesting a JSON specification of: team name, Warden (name,
description, authority scope), workers (same structure), delegation
strategy, and reasoning. The output is parsed, validated (must have a
Warden and at least two workers), and used to construct a complete
\texttt{HierarchicalWorkFlowGraph} with default escalation rules and
review mode.

This auto-build capability implements the Warden's agent lifecycle
management, the Warden dynamically spawns a team of specialist workers
tailored to each task, analogous to the ``spawning'' function described
in Section~\ref{sec:method_two}. In the Evolvian frontend, when a user
submits any task, the system automatically designs a specialist team
without requiring manual agent configuration, making the hierarchical
governance transparent to the end user.

\subsection{Ablation Study Design} \label{imp:five}

A four-configuration comparison isolates specialisation, hierarchy, and
evolution:

\begin{itemize}[itemsep=0.3em]

\item \textbf{Config A (Base EvoAgentX)}: Single agent, one LLM call.
Baseline for all RQs.

\item \textbf{Config B (Flat Pipeline)}: Three specialists chained
sequentially. No Warden, no review. Isolates \emph{specialisation}
(B vs A).

\item \textbf{Config C (Warden Hierarchy)}: Full team with delegation,
escalation, and review. 5--8 LLM calls. Isolates \emph{hierarchy}
(C vs B). Addresses \textbf{RQ1}.

\item \textbf{Config D (Hierarchy + Evolution)}: Config C plus
text-gradient prompt optimisation across generations. Isolates
\emph{evolution} (D vs C). Addresses \textbf{RQ3}.

\end{itemize}

\subsection{Evaluation Framework} \label{imp:six}

The evaluation framework ($\sim$2{,}700 lines, 8 files) provides a complete pipeline from benchmark loading through statistical analysis. Config runners implement A (single agent, regex-based answer extraction), B (flat sequential \texttt{WorkFlowGraph}), C (Warden hierarchy with graph deep-copying), and D (TextGrad evolution with generational rollback). \textbf{Answer extraction} (\texttt{answer\_extraction.py}) uses a lightweight LLM call (max 64 tokens, $T{=}0$) to produce concise answers from verbose multi-agent outputs, applied uniformly across all configurations. \textbf{Statistical comparison} (\texttt{compare\_results.py}) produces pairwise significance tests (Wilcoxon signed-rank, Welch's $t$-test), LaTeX \texttt{booktabs} tables, and \texttt{matplotlib} charts.

\subsection{Memory Layer and Persistence (RQ2)} \label{imp:seven}

The persistence function $M_{t+1} = f_{\text{persist}}(M_t, \Delta R_t)$
from Equation~3 is realised through three complementary mechanisms:

\begin{itemize}[itemsep=0.5em]

\item \textbf{Execution Traces ($\Delta R_t$)}: Every hierarchy event
(decomposition, delegation, worker execution, escalation, review,
revision, handoff) is recorded in the \texttt{ExecutionTrace} with
timestamps and metadata. Traces are serialised to JSON alongside
benchmark metrics, enabling post-hoc analysis of coordination dynamics.

\item \textbf{Quality Feedback Store}: The three-layer quality scoring
system (Section~4.9) persists assessments in the database across
sessions. Each workflow execution records proxy scores, LLM-judge
scores, and user ratings, constituting the persistent memory $M_t$
that informs subsequent workflow selection.

\item \textbf{WorkflowDNA Registry}: The Evolution Service maintains a
persistent registry of workflow configurations indexed by SHA-256
signature. Each entry accumulates performance statistics across
executions: average quality, cost, latency, success rate, and execution
count. New results $\Delta R_t$ consolidate into existing aggregate
metrics $M_t$, producing updated statistics $M_{t+1}$ that inform
future selection and mutation.

\end{itemize}

While agent-level episodic memory (as in MemGPT-style systems from
Section~2.4) was not implemented within scope, the combination of
these three mechanisms provides cross-session learning at the
orchestration level.

\subsection{Human-in-the-Loop Quality Feedback (RQ4)} \label{imp:eight}

Addressing RQ4, Evolvian implements a three-layer hybrid quality scoring
system (\texttt{core/\allowbreak runtime/\allowbreak quality\_evaluator.py}):

\begin{itemize}[itemsep=0.5em]

\item \textbf{Layer 1, Proxy Score}: A heuristic starting at 0.5 with
bonuses for output length, completion rate, speed, tool usage, and
assumption coverage. Provides instant feedback without LLM cost.

\item \textbf{Layer 2, LLM-as-Judge}: Scores on five dimensions,
Relevance, Completeness, Specificity, Coherence, and Quality,
each 0--10. The Coherence dimension directly addresses the metric
identified in Section~3.7.

\item \textbf{Layer 3, Human Rating (HITL)}: 1--5 star rating via
\texttt{POST /operations/\{id\}/rate}. The user observes the Warden's
execution via SSE streaming, evaluates the output, and provides a
rating that directly influences future workflow selection.

\end{itemize}

The hybrid formula prioritises human judgement: with user feedback
$0.6 \times \text{user} + 0.3 \times \text{LLM} + 0.1 \times \text{proxy}$;
without user feedback
$0.7 \times \text{LLM} + 0.3 \times \text{proxy}$.
Human ratings feed into the Evolution Service's fitness function,
meaning HITL supervision shapes the evolutionary trajectory,
not just correcting individual outputs but influencing which workflow
configurations are selected in future executions.

\subsection{Evolution Engine (RQ3)} \label{imp:nine}

The Evolution Engine is implemented in two complementary forms:

\begin{itemize}[itemsep=0.5em]

\item \textbf{Production Evolution Service} (\texttt{core/runtime/evolution.py}):
Implements $O_{\text{evo}}(G_t\allowbreak, E_t)$ from Equation~4 at the workflow level.
Maintains a population of \texttt{WorkflowDNA} configurations, each identified by SHA-256 signature.
The fitness function weighs quality (0.6), success rate (0.3), and efficiency (0.1) for quality-focused optimisation,
with an experience bonus (up to 0.2) rewarding well-tested workflows.
Mutation operators randomly add, remove, or reorder agents (rate 0.3).

\item \textbf{Benchmark Evolution Optimiser} (\texttt{run\_hierarchical\_evo.py}):
Implements text-gradient prompt evolution for Config~D.
Each generation: (1) collects failures from a training batch, (2) diagnoses root causes via an optimiser LLM,
(3) produces updated agent descriptions, (4) evaluates, (5) rolls back to the best generation.
This operates on semantic prompt content rather than model weights.

\end{itemize}

\subsection{Frontend Integration and Test Suite} \label{imp:ten}

The hierarchical execution system integrates into the Evolvian frontend via SSE streaming, rendering Warden decisions (decomposition, delegation, escalation, review) in real time for human oversight. 

The test suite comprises 99 tests across four files: \texttt{test\_team.py} (43), \texttt{test\_supervisor.py} (20), \texttt{test\_hierarchical\_graph.py} (18), \texttt{test\_execution.py} (18), using mock LLMs for API-independent validation. All 99 tests pass consistently.

\subsection{Datasets} \label{eva:datasets}

\subsubsection{HotPotQA}
\leavevmode\par

HotPotQA~\cite{yang2018hotpotqadatasetdiverseexplainable} (Yang et al., 2018; \url{https://hotpotqa.github.io/}) is an open-domain multi-hop QA benchmark containing 113{,}000 English Wikipedia-sourced question--answer pairs of two types: \textit{bridge} questions (requiring traversal across entity links, e.g.\ ``In what country was the director of \textit{Film X} born?'') and \textit{comparison} questions (requiring set-theoretic comparison of two entities). Each question is paired with 10 candidate paragraphs drawn from Wikipedia; exactly two are the gold supporting documents. Answers are short spans (names, dates, numbers, or brief phrases) enabling automatic F1 and exact-match evaluation. The full dataset has 90{,}564 training, 7{,}405 validation, and 7{,}405 test examples; we sample $n=20$ validation examples with seed 42 via EvoAgentX's built-in \texttt{HotPotQA} loader.

\textbf{Justification.} The multi-hop structure requires exactly the Warden's decompose-delegate-synthesise cycle: no single-hop lookup suffices. Retrieving supporting facts, reasoning across entities, and synthesising a final answer maps directly onto the three-worker chain (Retriever, Reasoner, Synthesiser), making HotPotQA the primary benchmark for evaluating RQ1. Metrics are token-level F1, exact match (EM), and answer-level accuracy (lenient substring match after answer extraction).

\subsubsection{MATH}
\leavevmode\par

The MATH benchmark~\cite{measuring_hendrycks_2021} (Hendrycks et al., 2021; \url{https://github.com/hendrycks/math}) contains 12{,}500 competition-style mathematics problems drawn from AMC 10/12, AIME, and MATHCOUNTS competitions. Problems span seven subjects (Prealgebra, Algebra, Number Theory, Counting \& Probability, Geometry, Intermediate Algebra, and Precalculus) across five difficulty levels (1 = easiest, 5 = hardest). Each problem has a structured step-by-step solution and a boxed final answer, enabling exact-match evaluation without human grading. We evaluate at Level~4--5 ($n=5$, pilot) and Level~2--3 ($n=20$, main evaluation) using the custom \texttt{MATHLevel23}/\texttt{MATHLevel45} wrappers that filter the EvoAgentX MATH loader by difficulty.

\textbf{Justification.} MATH provides symbolic, verifiable answers that allow automatic grading. Mathematical reasoning is a deliberate stress test for hierarchical decomposition: tight symbolic chains may be harmed by multi-agent handoffs (confirmed by our results), providing a clear boundary condition for RQ1. The difficulty shift from Level~4--5 to Level~2--3 was motivated by the pilot evaluation finding that Level~4--5 exceeded \texttt{gpt-4o-mini}'s capability ceiling (all multi-agent configurations scored 0.00). Metric is \textit{solve rate} (proportion of correctly answered problems).

It tests the Warden's governance on realistic, open-ended business research with explicit completeness requirements. Table~\ref{tab:enterprise-briefs} lists all 10 briefs.

\subsection{Escalation and Review Analysis} \label{eva:four}

\textbf{HotPotQA}: Escalation decreased from 60\% (pilot) to approximately 30\% (calibrated). When triggered, the Warden's direct access to all context typically produced higher-quality answers.

\textbf{MATH}: The calibrated Verifier reduced false INCORRECT flags from approximately 80\% to approximately 40\%, but escalation still degraded performance, because the Warden has the same mathematical limitations as workers.

\textbf{Enterprise}: The enterprise benchmark provides the richest governance data, with Config C averaging 1.9 review loops, 2.3 escalations, and 2.0 revisions per brief.

Three distinct behavioural modes emerged: \textit{efficient governance} (Briefs 3 and 4, 1--2 loops, 100\% alignment, quality 8.3--8.85, the Warden's single review pass catches missing sections and directs targeted revisions), \textit{active governance} (Briefs 1, 5, 7, and 10, 2--3 loops with iterative improvement trading latency for completeness), and \textit{diminishing returns} (Brief 6, 3 escalations but only 40\% alignment and 7.5/10 quality, governance cannot compensate when workers lack domain expertise). The sweet spot is 1--2 escalations with corresponding revisions; beyond 3 escalations, returns diminish.

\subsection{Evolution Analysis (RQ3)}\label{eva:five}

Config D's text-gradient evolution (\texttt{max\_steps=3}, \texttt{train\_k=5}) demonstrated correct mechanical operation, failure detection, diagnostic analysis, description mutation, and rollback all function as designed. On MATH, an intermediate generation showed 4$\times$ improvement (0.20 vs 0.05 baseline), but over-corrected in subsequent generations. On HotPotQA, evolution produced no net improvement. The rollback mechanism correctly restored the best-performing generation in both cases. \textbf{Conclusion}: evolution requires larger training batches ($n=5$ is insufficient) and more generations for adequate exploration.

\subsection{Human-in-the-Loop Analysis (RQ4)}\label{eva:six}

The three-layer quality feedback system provides: (1) real-time SSE visibility into Warden decisions (interpretability); (2) user ratings weighted at 60\% that shape evolutionary trajectory (error correction); (3) automated LLM-as-Judge quality assessment (coherence monitoring). The mechanism is fully implemented and operational; longitudinal experimental validation remains future work.
